# 管理员页与用户池绑定设计

## 背景

当前前端只有首页和单独的 `API Key` 管理页，右上角没有统一的用户入口。后端已经具备以下能力：

- `GET /api/admin/me` 可以判断当前用户是否为管理员
- `GET /api/cluster/gpu-overview` 可以返回节点、卡池类型和设备占用情况
- 节点池通过节点标签和污点同步器工作，独占池节点以 `genet.io/node-pool=non-shared` 识别

这次需求需要补齐两个能力：

1. 右上角新增个人入口，普通用户进入个人详情页，管理员进入管理员页
2. 管理员可以维护节点池和用户池，其中用户池绑定会影响后续 Pod 调度

## 目标

- 普通用户可以查看自己的个人信息和当前池归属
- 管理员可以在统一的管理员页中管理节点池、用户池和现有 API Key
- 用户被拖拽到共享池或独占池后，后续创建 Pod 只能调度到对应池
- 节点被拖拽到共享池或独占池后，系统继续复用现有节点标签与污点同步机制

## 非目标

- 不实现细粒度到具体节点的用户绑定
- 不实现多池、多租户、优先级队列
- 不修改已有 GPU 热力图的数据结构，只在管理页复用节点概览数据

## 方案选型

### 方案 A：统一管理员页

- 右上角使用统一用户菜单
- 普通用户跳转到 `/me`
- 管理员跳转到 `/admin`
- `/admin` 内部使用标签页承载概览、卡池管理、用户管理、API Key 管理

优点：

- 管理能力集中，后续扩展成本最低
- 现有 `API Key` 页面可并入，入口不会继续分散

缺点：

- 首次改造面比增加几个独立页面更大

### 方案 B：拆分多个独立页面

- 菜单中分别放个人页、卡池管理、用户管理、API Key 管理

优点：

- 单页实现更直接

缺点：

- 管理能力分散，后续维护体验差

## 选定方案

采用方案 A，建设统一管理员页。

## 信息架构

### 普通用户页 `/me`

展示：

- 用户名
- 邮箱
- 角色
- 当前池归属
- 当前说明文案，提示该池归属会限制 Pod 可调度节点范围

### 管理员页 `/admin`

标签页：

- `概览`
- `卡池管理`
- `用户管理`
- `API Key 管理`

#### 概览

- 当前管理员身份
- 共享池节点数 / 独占池节点数
- 共享池用户数 / 独占池用户数

#### 卡池管理

- 左右两列：`共享池`、`独占池`
- 节点卡片展示节点名、IP、设备类型、设备数
- 支持拖拽移动节点
- 拖拽后立即调用后端更新节点标签

#### 用户管理

- 左右两列：`共享池用户`、`独占池用户`
- 用户卡片展示用户名、邮箱、当前归属
- 支持拖拽移动用户
- 拖拽后立即调用后端更新用户池归属

#### API Key 管理

- 直接复用现有管理页能力，迁移进新标签页

## 后端设计

### 节点池管理

复用现有节点池实现，不新增新的池定义系统。

- 共享池：节点不带 `genet.io/node-pool=non-shared`
- 独占池：节点带 `genet.io/node-pool=non-shared`

管理员更新节点池时：

- 拖到独占池：为节点设置 `genet.io/node-pool=non-shared`
- 拖到共享池：移除该标签

现有 `StartNodePoolTaintReconciler` 会继续把独占池节点同步为带污点的节点。

### 用户池绑定存储

新增一个管理员态存储，用于保存 `username -> poolType` 映射。

首选实现：Kubernetes `ConfigMap`

- 命名空间：使用与现有管理数据一致的管理命名空间
- 名称建议：`genet-user-pool-bindings`
- 内容结构：JSON 或单个键值中的 JSON 文档

示例：

```json
{
  "alice": {
    "poolType": "exclusive",
    "updatedAt": "2026-03-14T10:00:00Z",
    "updatedBy": "admin"
  }
}
```

默认规则：

- 未绑定用户默认进入 `shared`

这样兼容旧用户，不需要迁移脚本。

### 管理 API

在现有 `/api/admin` 下新增接口：

- `GET /api/admin/overview`
- `GET /api/admin/nodes/pools`
- `PATCH /api/admin/nodes/:name/pool`
- `GET /api/admin/users/pools`
- `PATCH /api/admin/users/:username/pool`

返回数据约束：

- 节点池列表返回前端拖拽所需最小字段
- 用户池列表返回用户身份和归属
- `poolType` 统一使用 `shared | exclusive`

### 用户列表来源

优先使用当前集群内 Genet 管理过的 Pod、命名空间、已有绑定记录来汇总用户列表。

具体来源按并集构成：

- 运行中和历史上仍存在的 Genet Pod 上的 `genet.io/user`
- 已存在的 `user-*` 命名空间名称
- 用户池绑定 `ConfigMap` 中已记录的用户

这能避免一开始必须接企业通讯录或外部用户目录。

### 创建 Pod 时的调度限制

在 `CreatePod` 流程中新增用户池解析逻辑：

1. 根据当前登录用户读取池归属
2. 若为 `exclusive`，将 Pod 限制到独占池节点
3. 若为 `shared`，将 Pod 限制到共享池节点

实现方式：

- 对共享池注入 `nodeSelector`，要求节点不存在 `genet.io/node-pool=non-shared`
- 对独占池注入 `nodeSelector` 或 `nodeAffinity`，要求节点带 `genet.io/node-pool=non-shared`
- 对独占池补充对应污点容忍，确保能调度到被污点隔离的节点

同时增加校验：

- 如果请求显式指定 `nodeName`，但该节点不属于当前用户池，直接返回错误

## 前端设计

### 右上角用户入口

首页右上角从“多个并列按钮”调整为：

- 常用操作按钮保留
- 增加用户头像/名称下拉菜单

菜单项：

- 普通用户：`个人详情`
- 管理员：`管理员页`

为了避免入口过散，现有 `API Keys` 顶部按钮删除，统一收敛到管理员页。

### 视觉风格

延续当前玻璃拟态和浅/深色主题体系，不引入新的独立设计系统。

管理页强调：

- 左右双栏池面板
- 明确的共享池 / 独占池色彩区分
- 拖拽状态高亮
- 移动端下退化为纵向堆叠

### 拖拽实现

优先使用浏览器原生 HTML5 drag and drop，避免为两列拖拽引入额外依赖。

原因：

- 需求只需在两个池之间移动
- 现有项目依赖较轻，没必要为简单拖拽引入大型库

## 错误处理

- 管理接口失败时，保留原位置并提示错误消息
- 用户池绑定缺失时回退到 `shared`
- 节点更新冲突时提示刷新重试
- 指定节点与用户池不匹配时，创建 Pod 返回明确错误

## 测试策略

### 后端

- 管理接口鉴权测试
- 用户池 `ConfigMap` 读写测试
- 节点池标签更新测试
- 创建 Pod 时按用户池注入调度约束测试
- 指定非法池节点时拒绝创建测试

### 前端

- 右上角菜单按管理员/普通用户分流测试
- 管理员页标签与数据加载测试
- 卡池拖拽后触发更新测试
- 用户拖拽后触发更新测试
- 普通用户页显示当前池归属测试

## 风险与取舍

- 用户列表来自集群内痕迹数据，不等价于完整企业用户目录；这是有意的范围控制
- 共享池“排除独占标签”在 Kubernetes 原生 `nodeSelector` 中表达能力有限，实际实现更适合用 `nodeAffinity`
- 独占池节点已有污点，若不同时加容忍，将导致用户被绑定到独占池后无法成功调度

## 落地顺序

1. 增加用户池绑定存储与管理 API
2. 在创建 Pod 链路中接入用户池调度约束
3. 新增右上角用户入口与个人页
4. 新增统一管理员页并迁移 API Key 管理
5. 补齐前后端测试并验证回归
