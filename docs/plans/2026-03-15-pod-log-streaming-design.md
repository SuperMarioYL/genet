# Pod Log Streaming Design

## 背景

当前 `genet` 详情页的“实时日志”通过定时 HTTP 刷新实现。即使后端只取最近一段日志，前端仍然会周期性整段覆盖，导致两个问题：

- 性能差：每次刷新都重复拉取和重渲染已有日志
- 体验差：日志增长后，页面无法稳定保留连续上下文

用户要求将“当前日志”改成真正的日志流，同时保证性能；“之前日志”继续保持普通查询模式。

## 目标

- “当前日志”首次进入时先展示最近一段历史日志
- 历史日志展示后，立刻切换到流式追加，不再轮询整段日志
- 长时间打开页面时仍然可控，不因日志无限增长拖垮前端
- “之前日志”保持现有手动查看模式

## 非目标

- 本次不改“之前日志”为流式
- 本次不引入独立日志页面
- 本次不实现日志搜索、过滤、多容器切换

## 方案对比

### 方案 A：继续 HTTP 轮询，做增量拉取

优点：
- 改动较小

缺点：
- 仍然是轮询，不是真正日志流
- 需要维护游标和补偿逻辑

### 方案 B：SSE

优点：
- 单向推送，协议简单

缺点：
- 当前项目已经有 WebSocket 依赖和现成模式，复用度不如 WebSocket
- 前后端现有测试模式也更接近 WebSocket

### 方案 C：WebSocket 日志流

优点：
- 真正实时追加
- 可复用现有 `gorilla/websocket`、前端 `WebSocket` 和 URL 转换逻辑
- 后续如果要加状态消息或重连控制，扩展空间更大

缺点：
- 需要新增日志流端点和前端连接生命周期管理

## 结论

采用方案 C：仅对“当前日志”引入 WebSocket 日志流。

## 后端设计

新增 Pod 日志流端点：

- `GET /api/pods/:id/logs/stream`

行为：

- 认证后校验 Pod 是否存在
- 通过 Kubernetes `PodLogOptions{Follow: true, TailLines: N, Timestamps: true}` 建立日志流
- 服务端逐行读取日志，提取每行时间戳和正文
- 通过 WebSocket 发送 JSON 消息，例如：
  - `{"type":"chunk","content":"...","cursor":"2026-03-15T12:00:00.123456Z"}`
  - `{"type":"error","message":"..."}`

同时保留现有 `GET /api/pods/:id/logs`，并扩展：

- 支持 `tailLines`
- 支持可选 `since`
- 返回 `logs` 以及可选 `cursor`

这样“当前日志”首次加载历史时可以拿最近一段，连接断开后也可以按 `since` 做一次补齐。

## 前端设计

日志页拆成两条路径：

### 当前日志

- 首次进入时调用普通日志接口，获取最近一段历史
- 建立 WebSocket 连接，收到 `chunk` 后仅追加新日志
- 维护 `cursor`，用于断线重连前补齐缺失日志
- 保留缓冲上限，只保存最近若干行，避免长时间运行后 DOM 失控
- 跟随最新日志逻辑继续保留；用户手动上滑后只停止自动贴底，不停止日志接收

### 之前日志

- 继续走普通 `GET /logs?previous=true`
- 不建立流式连接
- 保持手动刷新

## 性能策略

- 首次历史日志仅取最近有限行数，例如 200 行
- WebSocket 仅发送新增日志，不重复发送历史内容
- 前端只追加，不整段覆盖
- 前端日志缓冲限制为最近 3000 行，超过后裁剪最早内容
- 仅当“当前日志”tab 处于激活状态时建立连接；切走 tab 或离开页面立即关闭

## 失败与重连

- WebSocket 意外断开时，前端先用当前 `cursor` 补拉一次缺失日志，再重连
- 如果补拉失败，保留已显示日志并提示连接状态异常
- 用户可手动点击“重新连接”

## 测试策略

后端：

- 验证日志流端点成功升级并发送日志消息
- 验证普通日志接口支持 `tailLines` 和 `since`

前端：

- 验证“当前日志”首次先加载历史，再建立 WebSocket
- 验证收到流消息后只追加新日志
- 验证切到“之前日志”时不会建立 WebSocket
- 验证超出缓冲上限时会裁剪旧日志
